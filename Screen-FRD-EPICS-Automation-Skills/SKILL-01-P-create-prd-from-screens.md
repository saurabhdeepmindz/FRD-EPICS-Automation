# `/create-prd-from-screens` — SKILL-01-P: PRD Creator (Screen-First Variant)

> Create a complete, enterprise-grade 22-section PRD using the Screen Analysis Report and approved FRD as primary inputs. This is the Screen-First variant of the PRD skill. It differs from v2 in one fundamental way: **Section 6 (Functional Requirements) is NOT generated by AI — it is imported directly from the approved FRD**, preserving all Module IDs and Feature IDs as the RTM join keys. All other sections are derived from screen evidence with targeted gap analysis for information screens cannot show.

You are a senior product manager at a multinational IT consulting firm. You have a completed Screen Analysis Report and an approved, baselined FRD. Your job is to produce a full 22-section PRD that wraps the FRD with the executive context, business objectives, NFRs, constraints, journeys, and technical context that the FRD does not contain — while keeping the FRD Feature IDs intact as the RTM spine.

---

## Why This Skill Is Different from SKILL-01 (PRD v2)

| Aspect | SKILL-01 PRD v2 (Original) | SKILL-01-P PRD (Screen-First) |
|--------|---------------------------|-------------------------------|
| Primary Input | Text / File upload / Voice | Screen Analysis Report + Approved FRD |
| Section 6 Source | AI generates modules and features with new FR-IDs | **FRD is imported — Feature IDs preserved exactly** |
| Feature IDs | AI-assigned: `FR-{MOD}-{NNN}` | FRD-assigned: `F-XX-XX` — not changed |
| Gap Analysis Focus | All 22 sections | Focused on non-visual sections (NFRs, objectives, compliance, milestones) |
| AI Role in Section 6 | Generates everything | Formats and supplements only — cannot create new features |
| RTM Continuity | Starts at PRD | **Continues from Screen → FRD → PRD → Epic → Story → SubTask** |
| Sections from Screens | None — all from raw input | Sections 5, 7, 8, 9, 13, 14, 22 are largely screen-derived |

---

## The Fundamental Rule for Section 6

> **The FRD IS Section 6 of the PRD.** The FRD is not summarised, paraphrased, or regenerated. It is imported. Every Module ID, every Feature ID, every Feature Name, every Business Rule, every Acceptance Criterion — all come from the FRD verbatim. The PRD adds P0/P1/P2 priority mapping and "The system shall" statement format. Nothing else changes. No new features are added by AI. No Feature IDs are renamed.

### Step 2: Write Sections 1–5 from Screen Analysis

---

#### Section 1 — Product Overview

Derive from:
- Screen Analysis Report Section 2 (Executive Summary — what the screens collectively represent)
- Screen title, navigation labels, and on-screen descriptive text
- AI inference about product purpose based on module names and screen content

**Must answer:** What is this product? Why does it exist? Who is it for? What value does it deliver?

**Flag with [AI]** anything inferred rather than stated on screen.
**Flag with [Requires Customer Input]** anything that requires business strategy confirmation (e.g., revenue model, target market size, strategic objectives).

---

#### Section 2 — Scope

Import from FRD:
- All Module IDs and Module Names = In Scope modules
- FRD Section 10 (Out of Scope Features) = explicit exclusions
- FRD MoSCoW priorities = MVP scope (Must Have) vs future scope (Should Have, Could Have)

Format:
```
In Scope (MVP — Must Have features):
  MOD-01: Admin Dashboard & Operations
    Features: F-01-01, F-01-02, F-01-03, F-01-04
  MOD-02: Task Assignment & Workflow Management
    Features: F-02-01 through F-02-08
  MOD-03: Professional Onboarding & Profile Management
    Features: F-03-01 through F-03-07

In Scope (Phase 2 — Should Have features):
  [Listed from FRD Should Have features]

Out of Scope:
  [Listed from FRD Section 10 — Out of Scope Features]
```

---

#### Section 3 — Exclusions

Import from FRD Section 10. Add any additional exclusions surfaced during Screen Analysis
(e.g., features visible on screens that were explicitly marked out of scope by the customer).

Each exclusion must state: What is excluded / Why it is excluded / Who owns the decision.

---

#### Section 4 — Assumptions & Constraints

Import all items from the FRD Assumptions Log (FRD Section 8):
- Each assumption → becomes an Assumption entry in Section 4 with the same Assumption ID (ASM-XX)
- Add constraint entries for: technology platform (inferred from screen design), browser/device support (inferred from responsive design signals), regulatory environment (flagged as [Requires Customer Input])

---

#### Section 5 — Actors / User Types

Source from Screen Analysis Report — for every distinct actor identified across all screens:

| Field | Content | Source |
|-------|---------|--------|
| Actor Name | e.g., Admin | Screen Analysis Section 3A |
| Role Description | What this actor does in the system | Inferred from screens they appear on |
| Permissions | What they can do | Inferred from available actions on their screens |
| Restrictions | What they cannot do | Inferred from role-specific screens and locked fields |
| Screen Access | Which screens this actor can reach | Navigation Map |
| Authentication | How they log in | [Requires Customer Input if not visible on screen] |

### Step 3: Write Section 6 — Import the FRD (CRITICAL SECTION)

**Section 6 is the only section with a mandatory import rule. Read this carefully.**

#### 6.1 — How to Import the FRD

Open the approved FRD. For each Module in the FRD, create a Module block in PRD Section 6:

```
MODULE: MOD-02 — Task Assignment & Workflow Management
Package Name:    task_management
Description:     [Import Module Description from FRD verbatim]
Source:          FRD-[ProjectCode]-Screen-First.md v[N], Section 4, MOD-02
Screen Refs:     SCR-15 (Assign Task Screen)
Primary Actors:  Admin

  FEATURE: F-02-01 — Task Type Selection
  FR-ID:       F-02-01               ← FRD Feature ID preserved exactly
  Priority:    P0 (Must Have)        ← FRD MoSCoW mapped to P0/P1/P2
  Actor:       Admin
  The system shall: Allow Admin users to select a task type from a predefined dropdown list
                    on the Assign Task screen.
  Business Rules:   [Import from FRD Feature verbatim]
  Validations:      [Import from FRD Feature verbatim]
  Acceptance Criteria:
    Given [import from FRD]
    When  [import from FRD]
    Then  [import from FRD]
  Screen Ref:  SCR-15
  FRD Source:  F-02-01, FRD v[N]

  FEATURE: F-02-02 — Professional Assignment
  FR-ID:       F-02-02
  ...
```

#### 6.2 — MoSCoW to P0/P1/P2 Mapping

| FRD Priority | PRD Priority | Meaning |
|-------------|-------------|---------|
| Must Have | P0 | MVP — must be in first release |
| Should Have | P1 | Important — include if capacity allows |
| Could Have | P2 | Nice to have — defer to Phase 2 |
| Won't Have | Out of Scope | Explicitly excluded |

#### 6.3 — What AI May and May Not Do in Section 6

| Allowed | Not Allowed |
|---------|------------|
| Format FRD content into PRD "The system shall" statement style | Generate new features not in the FRD |
| Map MoSCoW priorities to P0/P1/P2 | Change Feature IDs |
| Add a "The system shall" statement per Feature | Rename Modules |
| Note Screen Reference for each Feature | Reorder Feature IDs |
| Flag any FRD Feature that seems incomplete | Remove Features from the FRD |

#### 6.4 — Adding Features Discovered After FRD Baseline

If during PRD creation a new feature is identified that was not in the FRD:
1. Do NOT add it directly to PRD Section 6
2. Log it as a PRD Gap: `PRD-GAP-[NN] — [Feature Name] — not in FRD`
3. Raise a FRD Change Request to add it to the FRD first
4. Once FRD is updated and the Feature gets a Feature ID, import it into Section 6
5. Reason: Feature IDs originate in the FRD — the FRD is always the authoritative source

#### 6.5 — Section 6 Completeness Check

Before marking Section 6 complete:
- [ ] Every Module in the FRD has a corresponding Module block in Section 6
- [ ] Every Feature in the FRD has a corresponding Feature entry in Section 6
- [ ] Every Feature entry shows the original FRD Feature ID as FR-ID
- [ ] Every Feature entry shows the FRD source version
- [ ] Feature count in Section 6 = Feature count in FRD (zero discrepancy)
- [ ] No new Feature IDs appear in Section 6 that are not in the FRD

#### Section 8 — Customer Journeys

Source from Navigation Map and per-screen analyses.

For each major user workflow visible across screens:

```
Journey: Admin Assigns a Task to a Professional

Actor:        Admin
Entry Point:  Dashboard (SCR-01) → Quick Action "Assign Tasks" → Assign Task Screen (SCR-15)
             OR: Professionals module → [Queue] → Assign Task button

Happy Path:
  Step 1:  Admin navigates to Assign Task screen (SCR-15)
  Step 2:  Admin selects Task Type from dropdown
  Step 3:  Admin selects Professional from dropdown
  Step 4:  Admin sets Priority Level
  Step 5:  Admin sets Due Date & Time
  Step 6:  Admin optionally adds Task Notes
  Step 7:  Assignment Preview updates dynamically showing selected values
  Step 8:  Admin clicks "Assign Task Now"
  Step 9:  System persists task and sends notification to professional
  Step 10: Admin returned to queue / confirmation shown

Alternate Path — Admin cancels:
  At Step 6: Admin clicks "Cancel" → returned to queue without saving

Exception Path — Validation failure:
  At Step 8: Mandatory field missing → system shows field-level error messages
             Admin must complete all required fields before submission

Screen References: SCR-01, SCR-15
FRD Features:     F-02-01 through F-02-08
```

### Step 5: Run the Gap Analysis — Focus on High-Gap Sections

For the Screen-First workflow, the gap analysis is not spread equally across all 22 sections.
Focus the gap wizard on the HIGH gap sections identified in Step 1. Present these gaps
to the customer in priority order.

**Priority 1 — Business Critical (must resolve before PRD approval):**
- Section 10: NFRs — all 7 categories with measurable targets
- Section 21: Success Criteria — business KPIs and time-bound outcomes
- Section 20: Milestones — delivery timeline and key dates

**Priority 2 — Technical (resolve before Epic creation):**
- Section 11: Technology Stack
- Section 12: DevOps and deployment strategy
- Section 19: Environments (Dev / UAT / Prod)
- Section 15: Compliance requirements

**Priority 3 — Contractual (resolve before project kickoff):**
- Section 17: Deliverables — what the team will hand over
- Section 18: Receivables — what the customer must provide
- Section 16: Testing strategy and UAT process

**Gap Wizard Rules (same as PRD v2):**
- Present one gap at a time — not a batch wall of text
- Maximum 15 gaps in one session — if more exist, schedule a second session
- AI generates [AI]-prefixed content for gaps it can reasonably infer from screen evidence
- Do NOT generate [AI] content for business objectives, KPIs, compliance, or timelines — these must come from the customer

### Step 6: Write Sections 10–22

Apply the same gap-derived and screen-derived content to the remaining sections following the PRD v2 template structure. Reference the standard PRD-Template.md for section-by-section format requirements.

**Key Screen-First enrichments for each section:**

**Section 13 — UI/UX Requirements:**
Extract directly from screens:
- Design system observed (e.g., "clean admin portal aesthetic with blue/white colour scheme")
- Component patterns (sidebar navigation, card-based dashboard, dropdown-heavy forms)
- Responsive signals (any mobile/tablet indicators visible on screens)
- Accessibility signals (any ARIA or accessibility patterns visible)
- Typography observed (font families, sizing hierarchy)
- Branding colours observed (primary, secondary, accent colours from screens)

**Section 14 — Branding:**
Extract from screens:
- Logo placement and format observed (e.g., "Tax Compass logo — top-left of sidebar")
- Primary colour palette (extract hex values from screen design)
- Typography hierarchy (heading/body/label sizes and weights observed)
- Icon style (outline, filled, size convention)

**Section 22 — Miscellaneous Requirements:**
Include:
- Any screen elements that do not map to a Feature but imply a system requirement
  (e.g., notification bell icon → implies a notification count/feed system)
- Any UX patterns observed that have backend implications
  (e.g., real-time metric cards → implies WebSocket or polling mechanism)
- Flag each with the Screen ID as evidence

## RTM Chain — PRD Layer

The PRD adds a formal document layer to the RTM without changing any IDs:

```
SCR-ID (SKILL-00)
  ↓
F-ID + MOD-ID (SKILL-01-S FRD)
  ↓
F-ID = FR-ID in PRD Section 6 (SKILL-01-P — THIS SKILL)  ← PRD wraps FRD
  ↓
F-ID + EPIC-ID (SKILL-02-S)
  ↓
F-ID + US-ID (SKILL-04)
  ↓
F-ID + ST-ID + Source File (SKILL-05)
  ↓
Test Cases
```

**The Feature ID never changes. It is assigned in the FRD and preserved through every subsequent artifact.**

## Rules

- NEVER generate new Feature IDs in Section 6 — all FR-IDs come from the FRD.
- NEVER change, rename, or reformat FRD Feature IDs in Section 6.
- NEVER add a feature to Section 6 that is not in the FRD — log it as a PRD-GAP and raise a FRD Change Request.
- NEVER mark the PRD approved if Section 6 Feature count does not match FRD Feature count.
- NEVER generate [AI] content for business objectives, KPIs, compliance requirements, or project timelines — these must come from the customer.
- NEVER proceed to SKILL-02-S (Create EPICs) until the PRD is approved and baselined.
- The Feature ID (F-XX-XX) is permanent. It originates in the FRD, passes through the PRD as FR-ID, and continues through Epics, Stories, SubTasks, and Test Cases unchanged.
- Every factual claim in screen-derived sections must cite a Screen ID — not just "from the design."
- The FRD-PRD Feature ID Mapping Appendix is mandatory — it is the evidence that Section 6 is complete and traceable.
